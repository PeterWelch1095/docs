Instead of polling the Handshake API on a schedule, webhooks let you receive real-time HTTP POST notifications whenever key events occur — such as a new application, a student profile update, or an event registration. Your server receives the notification within seconds of the change, so your downstream systems stay current without the overhead and latency of repeated polling.

<Note>
  Webhook support availability depends on your integration tier. Contact your Handshake account manager to confirm webhook access for your account.
</Note>

## Webhook payload format

When a tracked event occurs, Handshake sends an HTTP POST request to your registered endpoint with a JSON body. Every payload follows this structure:

```json
{
  "event_type": "application.created",
  "occurred_at": "2024-11-14T09:32:00Z",
  "data": {
    "id": "app_9f3kd82",
    "student_id": "stu_4a1bc99",
    "job_id": "job_7de302f",
    "status": "applied",
    "created_at": "2024-11-14T09:32:00Z",
    "updated_at": "2024-11-14T09:32:00Z"
  }
}
```

| Field | Description |
|---|---|
| `event_type` | The type of event that occurred (see below) |
| `occurred_at` | ISO 8601 timestamp of when the event occurred in Handshake |
| `data` | The full resource object affected by the event |

**Supported event types:**

| Event type | Triggered when |
|---|---|
| `application.created` | A student submits an application to a job |
| `application.updated` | An application's status changes |
| `student.updated` | A student updates their profile |
| `event.attendance.created` | A student registers for an event |
| `event.attendance.updated` | An attendance record is updated (e.g. check-in marked) |

## Verifying webhook signatures

Handshake signs every webhook payload using a shared secret. The signature is included in the `X-Handshake-Signature` HTTP header as an HMAC-SHA256 hex digest of the raw request body.

Always verify this signature before processing the payload. If verification fails, discard the request — it may have been tampered with or did not originate from Handshake.

```python
import hashlib
import hmac


WEBHOOK_SECRET = "<YOUR_WEBHOOK_SECRET>"  # Provided by Handshake


def verify_signature(raw_body: bytes, signature_header: str) -> bool:
    """
    Verify that the incoming webhook payload was signed by Handshake.

    Args:
        raw_body: The raw (undecoded) request body bytes.
        signature_header: The value of the X-Handshake-Signature header.

    Returns:
        True if the signature is valid, False otherwise.
    """
    expected = hmac.new(
        key=WEBHOOK_SECRET.encode("utf-8"),
        msg=raw_body,
        digestmod=hashlib.sha256,
    ).hexdigest()

    # Use compare_digest to prevent timing attacks
    return hmac.compare_digest(expected, signature_header)
```

## Handling webhooks

A reliable webhook handler does three things: verifies the signature, acknowledges receipt immediately with an HTTP 200, and processes the event asynchronously so the response is never delayed by downstream work.

**Steps:**

1. **Parse and verify** — read the raw body, check the `X-Handshake-Signature` header, and reject invalid requests with a `400` response.
2. **Return HTTP 200 immediately** — do this before any processing. Handshake considers any non-2xx response a delivery failure and will retry.
3. **Process asynchronously** — hand the payload off to a background queue or worker so that slow database writes or external API calls don't block your response.

```python
import hashlib
import hmac
import json
from flask import Flask, request, abort
from your_queue import enqueue_task  # Replace with your actual task queue

app = Flask(__name__)
WEBHOOK_SECRET = "<YOUR_WEBHOOK_SECRET>"


@app.route("/webhooks/handshake", methods=["POST"])
def handshake_webhook():
    raw_body = request.get_data()
    signature = request.headers.get("X-Handshake-Signature", "")

    # Step 1: Verify signature
    expected = hmac.new(
        key=WEBHOOK_SECRET.encode("utf-8"),
        msg=raw_body,
        digestmod=hashlib.sha256,
    ).hexdigest()

    if not hmac.compare_digest(expected, signature):
        abort(400, "Invalid signature")

    # Step 2: Return 200 immediately
    payload = json.loads(raw_body)

    # Step 3: Hand off to background processing
    enqueue_task("process_handshake_event", payload)

    return "", 200


def process_handshake_event(payload: dict):
    """Background worker — called outside the request cycle."""
    event_type = payload["event_type"]
    data = payload["data"]

    if event_type == "application.created":
        handle_new_application(data)
    elif event_type == "student.updated":
        handle_student_update(data)
    elif event_type == "event.attendance.created":
        handle_event_registration(data)
    else:
        print(f"Unhandled event type: {event_type}")
```

## Retry behavior

If your endpoint returns a non-2xx status code — or does not respond before the request times out — Handshake treats the delivery as a failure and retries automatically. Retries use exponential backoff, so the interval between attempts increases with each failed delivery.

Because the same event may be delivered more than once, your handler must be **idempotent**: processing the same payload twice should produce the same result as processing it once. The simplest way to achieve this is to key your database writes on the resource `id` field in the `data` object and use an upsert strategy rather than a plain insert.

```python
def handle_new_application(data: dict):
    """Idempotent handler — safe to call multiple times with the same payload."""
    db.upsert(
        "applications",
        record=data,
        conflict_key="id",  # Upsert on the resource ID
    )
```

<Warning>
  Always verify the webhook signature before processing the payload. Never trust unverified webhook data. A missing or invalid `X-Handshake-Signature` should cause your handler to return a `400` response and discard the request entirely.
</Warning>
